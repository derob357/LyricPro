# Brief for Jim — LyricPro Ai: Path to Production

Hey Jim — quick check-in on where LyricPro Ai stands and what's between us and launch. The good news is the game itself works end-to-end: we've got the new trivia-show-style gameplay screen rendering in the purple/cyan brand palette, a three-stage round (main question → artist bonus → year bonus) that all shares a single timer, a buzz-in mechanic where the first player's key-press desaturates the screen for the others, and a song library that just grew from 46 to **900 songs across eight genres** (Hip Hop, R&B, Pop, Rock, Country, Gospel, Soul, Jazz — each genre has 100+ songs, with iconic chart hits mixed with deeper cuts reserved for the high-difficulty rounds). On the security side I've already closed three real holes: a privilege-escalation bug where any logged-in user could grant themselves a paid "Elite" tier via a direct API call, a crash loop where an unreplaced analytics template variable was taking down the dev server, and an unbounded loop in the year-distractor generator. On top of that, I hardened the Stripe payment layer with four fixes — **webhook idempotency** (so Stripe re-delivering an event can't double-charge or double-credit someone), **proper refund handling** that reverses entry-fee participation and flags subscription refunds for review, **subscription renewal** that now actually extends the user's paid period on each `invoice.paid` event, and **rate limiting** on checkout creation (capped at 10 attempts per 10 minutes per user) so nobody can abuse the Stripe Checkout endpoint.

What's still ahead to ship to production: (1) you'll need to give me a real Stripe **test-mode secret key** so I can run an end-to-end purchase test and then swap it for a production key at launch; (2) we need to choose between **Manus OAuth** (already wired) or swap in Google/Apple sign-in for the App Store path; (3) the monetization UI still needs the entry-fee selector wired into the game-setup screen and a prize-payout flow via Stripe Connect (Stripe's payout plumbing is already written, just not invoked when a game ends); (4) we need a real production database and deployment target — the MySQL schema and migrations are drizzle-managed, so spinning up a hosted instance on Railway, PlanetScale, or RDS takes about an hour. For the **mobile app** question, my strong recommendation is Path 1: ship mobile as a free/trial-only experience (no in-app purchases) and keep all paid subscriptions + entry fees web-only — same approach Spotify and Netflix use to avoid Apple's 30% cut and the much harder "real-money contest" App Store rules that would otherwise likely kill iOS approval. We'd wrap the existing React app with Capacitor (about a day of setup), get a buildable Xcode and Android Studio project, and submit to both stores. Roughly my estimate from here: **1 week** to finish the monetization plumbing and test Stripe end-to-end, **1 week** for Capacitor wrapping + store assets + submission, **1–2 weeks** waiting on App Store / Play review. Let me know if that sequencing works for you and I'll book the time.
